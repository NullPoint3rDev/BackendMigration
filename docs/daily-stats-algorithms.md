# Алгоритмы суточной статистики сварочного аппарата

Расчёты реализованы в `WeldingMachineDailyStatsService` (пересчёт суток `recomputeDayAndSave`)
с использованием `WeldingStateDurationUtil` и `MonitorActivityClassifier`.

## 1. Расчёт таймеров активности за сутки

Считаются четыре таймера (в миллисекундах): `off` (выключен), `error` (авария/ошибка),
`on` (включён/дежурит), `welding` (сварка).

**Границы суток.** Начало суток — `statDate` в 00:01 (`dayStart`, не с полуночи).
Конец — начало следующих суток по таймзоне `Europe/Moscow` (`dayEnd`). Если считаются
текущие сутки, конец обрезается до текущего момента (`effectiveEnd = min(now, dayEnd)`).

**Загрузка данных.** Из БД берутся все состояния аппарата (`WeldingMachineState`) за
интервал `[dayStart, dayEnd)`, сортируются по `dateCreated`. Для каждого состояния
подгружаются его параметры: текст состояния («Состояние аппарата» / `WeldingMachineState` /
`State.WeldingMachineState`) и ток (`State.I` / «Ток» / `Current`).

**Проход по состояниям.** Для каждого состояния:

1. Вычисляется пересечение его длительности с окном `[dayStart, effectiveEnd)` — функция `overlapDurationMs`:
   - длительность состояния = `stateDurationMs`, если задана; иначе = зазор до `dateCreated` следующего состояния;
   - у **последнего** состояния хвост до `now` намеренно не добавляется (`openEndIfLast = null`) — иначе таймеры «скачут» на последнем опросе;
   - берётся длина именно перекрытия отрезка состояния с окном суток.
   - Если перекрытие ≤ 0 — состояние пропускается.
2. Состояние классифицируется в один из режимов (`MonitorActivityClassifier.classify`) по приоритету:
   - **welding** — текст содержит «сварка»/`welding`/`weld`/«сварочн», ИЛИ статус = `Welding`, ИЛИ текста нет, но ток > 1 А;
   - **error** — текст содержит «авария»/«ошибка»/`error`/`emergency`/`failure`, ИЛИ `errorCode` > 0 (или нечисловой), ИЛИ статус = `Error`;
   - **off** — текст содержит «выключ»/`off`/`offline`/«не в сети»/«дежур»/`standby`, ИЛИ статус = `Offline`;
   - **on** — «ожидан»/`waiting`/«включ»/`idle`/`ready`, и как режим по умолчанию.
3. Длительность перекрытия прибавляется к соответствующему таймеру.

**Замощение суток.** Чтобы сумма таймеров равнялась прошедшему времени `[00:01, effectiveEnd]`,
непокрытое телеметрией время относится к `off` («Выкл./не в сети»):
- голова `[00:01 → первый пакет дня]`;
- хвост `[последний покрытый момент → effectiveEnd]` (последний покрытый момент = `dateCreated`
  последнего состояния + его `stateDurationMs`, обычно = само `dateCreated`, т.к. хвост до `now`
  намеренно не тянется — иначе таймеры «скачут»);
- весь интервал суток, если пакетов не было вовсе.
Штатные короткие зазоры между опросами уже учтены через «зазор до следующего состояния».

**Итог.** Суммы `offMs / errorMs / onMs / weldingMs` сохраняются в `WeldingMachineDailyStats`.
В DTO поле `standbyMs` из БД отдаётся как `errorMs` (переименование), а `standbyMs` наружу = 0.

## 2. Расчёт суточного расхода газа

Расход газа считается по накопительному счётчику `Core.GasConsumptionSincePowerOn`
(литры «с момента включения») методом суммирования положительных дельт.

**Данные.** Берутся состояния за расширенное окно `[dayStart − 1 день, dayEnd)` (лишние
сутки нужны, чтобы найти «базовое» значение счётчика на начало суток). Для каждого
состояния подгружается значение накопительного счётчика газа.

**Проход по состояниям (по возрастанию времени):**

- Если состояние **раньше** `dayStart` — его значение запоминается как `baselineAtDayStart`
  и как `lastCumulative` (это то, что накопилось до начала суток), в сумму не идёт.
- Если состояние в пределах суток:
  - если `baseline` ещё не задан — первое значение становится `baseline`;
  - к сумме прибавляется дельта между `lastCumulative` и текущим значением (`sumGasCumulativeDelta`):
    - если счётчик **вырос** (`current ≥ last`) → дельта = `current − last`;
    - если счётчик **упал**, но это распознано как **сброс питания** → дельта = само `current` (счётчик обнулился и начал заново);
    - иначе (мелкая просадка от обрыва связи) → дельта = 0.
  - `lastCumulative` обновляется до `current` только при росте или реальном сбросе (иначе игнорируется провал).

**Определение сброса счётчика** (`isGasCounterPowerOnReset`): считается сбросом, только
если падение ≥ 100 л И новое значение < 50% от предыдущего. Это отсекает мелкие глюки
связи от настоящего обнуления при перезагрузке аппарата.

**Итог.** `gasConsumptionL` = сумма дельт (округление до 3 знаков), плюс сохраняется
`gasBaselineAtDayStartL`. Если данных счётчика нет — расход 0.

## 3. Расчёт суточного расхода проволоки

Параметр «Расход проволоки» (`WIRE_PARAM`) с аппарата — это **накопительная длина поданной
проволоки в метрах «с момента включения»** (uint32 из пакета Core, декодированный как IEEE-float),
а НЕ мгновенная скорость. Значение монотонно растёт при сварке и стоит в простое, обнуляется
при перезагрузке аппарата. Поэтому расход считается по **дельтам счётчика**, как газ.

**Данные.** Состояния за сутки `[dayStart, dayEnd)` по времени. Для каждого подгружается
значение счётчика «Расход проволоки» (обычный маппинг + fallback на нативный `COALESCE`).

**Константа.** Линейная плотность проволоки `wireLinearDensityKgPerMeter` = 0.0089 кг/м
(сталь Ø1.2 мм, из конфигурации `report.wire.linear-density-kg-per-meter`).

**Проход по состояниям (по возрастанию времени), `sumWireCumulativeMeters`:**

1. Для каждой пары последовательных значений счётчика берётся дельта.
2. Если счётчик **вырос** → прибавляем `current − last`.
3. Если счётчик **упал** (перезагрузка «с включения») → прибавляем само `current`.
4. Сумма положительных дельт = поданная за сутки длина проволоки (м).

Формула по сути:

```
масса (кг) = Σ положительных_дельт_счётчика (м) × плотность (кг/м)
```

**Итог.** Метры × плотность, округление до 5 знаков, сохраняется как `wireConsumptionKg`.
ponytail: плотность фиксирована под сталь Ø1.2 мм; апгрейд — таблица плотностей по
диаметру/материалу (в пакете есть `weldingWireDiameter`/`weldingMaterial`).

## Общие замечания

Пересчёт (`recomputeDay`) детерминирован и кэшируется в `WeldingMachineDailyStats`. Для
текущих суток при устаревшем кэше (старше 60 с) идёт синхронный пересчёт, для прошлых дат —
асинхронный с дебаунсом 15 с.
